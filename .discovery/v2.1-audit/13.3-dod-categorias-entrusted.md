# DoD por categoría ENTRUSTED+ · 9 plantillas v3.4

Plantillas reusables para inyectar `DEFINITION OF DONE`, `CRITERIOS DE ACEPTACIÓN OBSERVABLES` y `EDGE CASES` en cada SPEC v3.4. Cada plantilla se personaliza al output específico del prompt al aplicarse.

---

## 1. `análisis`

**DoD:** produce un análisis estructurado con (a) hechos extraídos del material con etiqueta de procedencia ({ADJUNTO|EXTRAIDO_HILO|WEB|CONOCIMIENTO|INFERENCIA}), (b) hallazgos numerados con confianza explícita 0.0-1.0, (c) recomendación priorizada con criterio operativo (impacto/esfuerzo/riesgo), (d) sección "Lo que NO se sabe" con `{VACIO_CRITICO}`s y propuesta de cierre.

**Criterios de aceptación observables (5):**
- Cada hallazgo lleva al menos 1 etiqueta de procedencia trazable.
- Recomendación final tiene criterio explícito de priorización (no opinión).
- Lista de `{VACIO_CRITICO}`s con propuesta de cómo cerrarlos.
- Confianza global ≥ 0.95 (o debilidad declarada con plan).
- Bloque INSIGHTS_PROACTIVOS al cierre con riesgos/oportunidades no obvias.

**Edge cases típicos (3):**
- *Material insuficiente* → entrega análisis parcial + lista explícita de gaps + qué dato cierra cada gap.
- *Material contradictorio* → enumera conflictos + lado más sustentado + dato decisor faltante.
- *Material PII detectado* → anonimiza antes de analizar; declara qué se redactó.

## 2. `generación`

**DoD:** produce el artefacto solicitado (texto/email/post/spec/script/etc.) con (a) longitud y formato dentro del rango pedido (±10%), (b) tono coherente con audiencia, (c) todos los inputs declarados utilizados o marcados como `{POR_CONFIRMAR}` con placeholder, (d) cero invenciones de citas, datos específicos o nombres reales sin marcar.

**Criterios de aceptación observables (5):**
- Longitud final dentro del rango ±10% del pedido.
- Tono verificable (registro, tutear/usted, jerga del dominio) coherente con audiencia declarada.
- Cada input de INPUTS aparece literal o etiquetado como autocompletado en el output.
- Cero claims factuales no etiquetados (números, fechas, citas).
- Bloque INSIGHTS_PROACTIVOS con 1-3 variantes alternativas o ángulos no pedidos.

**Edge cases típicos (3):**
- *Tono pedido ≠ audiencia natural* → entrega versión pedida + alerta del conflicto + versión recomendada.
- *Datos inventables* (estadísticas, citas) → marca con `{POR_CONFIRMAR}` + placeholder + sugerencia de fuente.
- *Restricción de longitud impide profundidad* → entrega versión + nota de qué se omitió + dónde recuperarlo.

## 3. `extracción`

**DoD:** produce una estructura de datos (tabla/lista/JSON/YAML) con (a) campos exactos declarados, (b) valor por celda con etiqueta de procedencia inline, (c) celdas faltantes marcadas `{VACIO_CRITICO}` o `{POR_CONFIRMAR}`, (d) reporte de cobertura (% completitud + N de celdas vacías).

**Criterios de aceptación observables (5):**
- Estructura final exacta (mismos campos, mismo orden).
- Cero celdas inferidas presentadas como hechos sin etiqueta.
- Reporte de cobertura cuantitativo (X/Y celdas completas, N% gap).
- Trazabilidad celda→fuente para auditoría.
- Bloque INSIGHTS_PROACTIVOS con duplicados detectados, anomalías, patrones cruzados.

**Edge cases típicos (3):**
- *Material parcial* → estructura completa + celdas vacías marcadas + plan para completarlas.
- *Datos contradictorios entre fuentes* → reporta conflicto en celda + valor más sustentado + bandera al usuario.
- *Schema ambiguo* → propone schema + valida con usuario antes de extraer masivo.

## 4. `agente`

**DoD:** produce un plan ejecutable de N pasos con (a) acción concreta por paso, (b) input/output declarado por paso, (c) precondición y postcondición observables, (d) handlers de error para los 3 fallos más probables, (e) estado memorizable entre pasos.

**Criterios de aceptación observables (5):**
- Cada paso tiene precondición y postcondición verificables.
- 3+ handlers de error declarados con criterio de aborto.
- Estado entre pasos especificado (qué guardar para paso N+1).
- Bloque INSIGHTS_PROACTIVOS con riesgos del flujo + paths alternativos + métricas de éxito.
- Plan ejecutable sin reabrir el contexto original.

**Edge cases típicos (3):**
- *Step bloqueado por dependencia externa* → declara gap + plan B + criterio de cuándo abortar.
- *Output del paso N no cumple precondición de N+1* → loop de corrección + límite de iteraciones + criterio de aborto.
- *Cambio de contexto entre invocaciones* → mecanismo de resumen para reanudar (memoria viva).

## 5. `automatización`

**DoD:** produce un blueprint operativo (flow / script / config) con (a) trigger declarado, (b) acciones secuenciales con conectores específicos, (c) variables tipadas, (d) manejo de errores con notificación, (e) gobernanza (logs, reintentos, escalación), (f) prueba mínima de aceptación.

**Criterios de aceptación observables (5):**
- Trigger explícito con condiciones de activación.
- Cada acción con conector específico (no genérico).
- Variables con tipo y default.
- 3+ casos de error con handler.
- Bloque INSIGHTS_PROACTIVOS con cuellos de botella, dependencias críticas, métricas a monitorear.

**Edge cases típicos (3):**
- *Conector externo no autenticado* → declara dependencia + plan de fallback + alerta.
- *Throttling / rate limit* → reintentos exponenciales + saturación de cola + escalación.
- *Datos inválidos* → validación antes de cada acción + dead-letter queue + notificación.

## 6. `investigación`

**DoD:** produce un reporte de investigación con (a) preguntas guía explícitas, (b) hallazgos con fuentes citables (mín 3 por hallazgo crítico), (c) síntesis ejecutiva, (d) brechas de conocimiento + propuesta de siguientes pasos, (e) confianza por hallazgo.

**Criterios de aceptación observables (5):**
- Mínimo 3 fuentes por hallazgo crítico (etiquetadas {WEB|ADJUNTO|CONOCIMIENTO}).
- Síntesis con confianza explícita 0.0-1.0.
- Lista explícita de brechas + cómo cerrarlas.
- Cero claims sin fuente o etiqueta.
- Bloque INSIGHTS_PROACTIVOS con preguntas no exploradas, hipótesis alternativas.

**Edge cases típicos (3):**
- *Fuentes contradictorias* → enumera conflicto + lado más sustentado + dato decisor.
- *Tópico fuera del knowledge cutoff* → declara explícitamente + sugiere búsqueda web (`ORQUESTACIÓN_RECURSOS: web=on`).
- *Material insuficiente* → reporta brecha + propone fuentes alternativas.

## 7. `estrategia`

**DoD:** produce un análisis estratégico con (a) framework aplicado correctamente (SWOT/Porter/BCG/etc.), (b) datos del caso integrados con etiqueta, (c) recomendación accionable con horizonte (corto/medio/largo), (d) priorización con criterio explícito, (e) riesgos + mitigaciones.

**Criterios de aceptación observables (5):**
- Framework aplicado completo (4 cuadrantes/fuerzas/etc., no parcial).
- Datos del caso integrados en cada celda del framework con etiqueta.
- Recomendación con horizonte temporal explícito.
- Priorización con criterio (impacto/esfuerzo/riesgo) cuantificado.
- Bloque INSIGHTS_PROACTIVOS con suposiciones cuestionables, alternativas no consideradas.

**Edge cases típicos (3):**
- *Datos del caso insuficientes para 1+ cuadrante* → marca celda como `{VACIO_CRITICO}` + propone cómo obtenerlo.
- *Framework no encaja al 100%* → adapta con justificación + alerta al usuario.
- *Recomendación con riesgo alto* → escenario + mitigación + plan de salida.

## 8. `multimedia`

**DoD:** produce especificación visual/audiovisual ejecutable en herramienta destino con (a) parámetros técnicos (resolución, duración, aspect ratio, etc.), (b) descripción de composición (sujeto/fondo/iluminación/cámara), (c) estilo declarado (referencias artísticas si aplica), (d) brief de copy/voz si aplica.

**Criterios de aceptación observables (5):**
- Parámetros técnicos completos para herramienta destino (Imagen/DALL-E/Veo/Sora/Midjourney/etc.).
- Composición descrita en mínimo 4 elementos visuales.
- Estilo declarado con referencias o atributos concretos.
- Cero ambigüedades sobre encuadre y aspecto ratio.
- Bloque INSIGHTS_PROACTIVOS con variantes de mood/encuadre, alternativas estilísticas.

**Edge cases típicos (3):**
- *Herramienta destino no especificada* → entrega versión genérica + parámetros típicos + alerta.
- *Restricciones de la herramienta* (no permite cierto contenido) → marca + sugiere reencuadre del prompt.
- *Estilo conflictivo con marca* → alerta + propone reconciliación + entrega versión alineada.

## 9. `procedimiento`

**DoD:** produce una plantilla / checklist / receta paso-a-paso con (a) precondición de inicio, (b) pasos numerados con verbo accionable + entregable observable por paso, (c) validación entre pasos, (d) postcondición de cierre, (e) plantilla reutilizable para casos similares.

**Criterios de aceptación observables (5):**
- Cada paso tiene verbo en imperativo + entregable verificable.
- Precondición y postcondición observables.
- Validación intermedia con criterio (no opinión).
- Plantilla portable (funciona en otros casos del mismo dominio).
- Bloque INSIGHTS_PROACTIVOS con anti-patrones, automatización futura, métricas.

**Edge cases típicos (3):**
- *Paso bloqueado por dato faltante* → marca `{VACIO_CRITICO}` + skip condicional + recuperación.
- *Caso fuera del scope del procedimiento* → declara excepción + fallback manual + log para refinar plantilla.
- *Postcondición no se cumple* → diagnóstico de por qué + corrección + retry.

---

## Reglas comunes a las 9 categorías

- **Cero invenciones**: cada hecho con etiqueta de procedencia.
- **Cero fricción innecesaria**: la IA infiere desde hilo/memoria/adjuntos antes de pedir.
- **Confianza ≥ 0.95** o debilidad declarada con plan de cierre.
- **Bloque INSIGHTS_PROACTIVOS** obligatorio al cierre (sugerencias + recomendaciones + riesgos + preguntas si aplica).
- **METADATA DE RAZONAMIENTO** al cierre con confianza global, fuentes, autocompletados, debilidades, próximo checkpoint, rigidez.
- **Standalone** + **encadenable** como insumo del paso siguiente del pipeline P.I.V.O.T.E.
